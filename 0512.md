## 一、帳號管理

---

### 1-1 管理買家帳號－事件條列式

| | |
|---|---|
| **二手買賣市場** | |
| **使用個案名稱** | 管理買家帳號 |
| **行為者** | 管理員 |
| **目標** | 管理員可查詢、檢視並管理買家帳號資訊，包含查看基本資料、交易紀錄與檢舉紀錄，並可執行帳號停權或解除停權等操作 |
| **前提** | 管理員已登入後台系統，且具備帳號管理權限 |
| **結束狀態** | 帳號資訊已更新，或相應處置已完成並記錄於系統 |

**一系列事件：**

正常程序—

1. 管理員進入後台「買家帳號管理」頁面，輸入關鍵字（如：帳號名稱、Email）進行搜尋，或依條件篩選買家清單。
2. 系統自「會員主檔」查詢符合條件之買家帳號，顯示帳號清單（含帳號狀態、註冊日期、交易次數等）。
3. 管理員點選特定買家帳號，進入帳號詳情頁面。
4. 系統顯示買家基本資料（Email、暱稱、聯絡方式）、交易紀錄、被檢舉紀錄及帳號狀態。
5. 管理員依需求執行操作：
   - 若執行**停權**：填寫停權原因，確認後系統更新帳號狀態為「停權中」，並發送通知信至買家信箱。
   - 若執行**解除停權**：確認後系統更新帳號狀態為「正常」，並發送解除通知信。
   - 若僅**查看資訊**：不執行異動，直接離開頁面。
6. 系統將操作紀錄寫入後台操作日誌。

例外狀況—

> ➢ 搜尋條件無符合帳號，系統提示「查無符合條件之買家帳號」。
>
> ➢ 管理員嘗試對已停權帳號再次執行停權，系統提示「此帳號已處於停權狀態」。
>
> ➢ 操作過程中連線逾時，系統提示「操作失敗，請重新整理頁面後再試」。

### 1-3 停權處理（賣家）－事件條列式

| | |
|---|---|
| **二手買賣市場** | |
| **使用個案名稱** | 停權處理（賣家） |
| **行為者** | 管理員 |
| **目標** | 管理員針對違規賣家執行帳號停權，限制其登入與交易行為，並同步處理其商品與進行中訂單 |
| **前提** | 管理員已進入特定賣家帳號詳情頁面，且已確認具備停權執行依據 |
| **結束狀態** | 賣家帳號狀態更新為「停權中」，相關商品與訂單已完成處置，通知已發送 |

**一系列事件：**

正常程序—

1. 管理員於賣家帳號詳情頁面點擊「執行停權」按鈕。
2. 系統彈出停權確認視窗，要求管理員填寫停權原因（必填）及停權期限（選填：永久停權或指定到期日）。
3. 管理員確認後送出。
4. 系統執行以下操作：
   - 更新賣家帳號狀態為「停權中」。
   - 批次下架該賣家所有「上架中」之商品（狀態更新為「強制下架」）。
   - 查詢該賣家所有「處理中」訂單，標記為需人工介入，並通知相關買家。
5. 系統發送停權通知信至賣家信箱，說明停權原因與申訴方式。
6. 系統將本次停權操作寫入後台操作日誌，記錄執行管理員帳號、時間戳記與停權原因。

例外狀況—

> ➢ 管理員未填寫停權原因即送出，系統提示「請填寫停權原因後再送出」。
>
> ➢ 系統批次下架商品時部分失敗，系統提示「部分商品下架失敗，請至商品管理頁面手動確認」，並列出失敗商品清單。
>
> ➢ 發送通知信失敗，系統提示「通知信發送失敗，請至通知管理手動補發」。

### 1-4 身分驗證審核（賣家）－事件條列式

| | |
|---|---|
| **二手買賣市場** | |
| **使用個案名稱** | 身分驗證審核（賣家） |
| **行為者** | 管理員 |
| **目標** | 管理員審核賣家提交之身分驗證文件，判斷是否核准賣家資格，並通知審核結果 |
| **前提** | 賣家已完成身分驗證文件上傳，驗證申請狀態為「審核中」；管理員已登入後台且具備審核權限 |
| **結束狀態** | 審核結果已更新至賣家帳號，通知已發送，操作已記錄 |

**一系列事件：**

正常程序—

1. 管理員進入後台「身分驗證審核」列表頁面，系統顯示所有狀態為「審核中」的驗證申請，按提交時間排序。
2. 管理員點選特定申請，進入審核詳情頁面。
3. 系統顯示賣家上傳之身分驗證文件（如：身分證正反面影像、銀行帳戶資料），及賣家基本資料。
4. 管理員逐一檢視文件，進行審核判斷：
   - 若**審核通過**：點擊「核准」，系統更新賣家身分驗證狀態為「已驗證」，開放完整賣家功能權限。
   - 若**審核不通過**：點擊「退回」，填寫退回原因（必填），系統更新狀態為「驗證未通過」，並通知賣家補件或重新申請。
5. 系統發送審核結果通知信至賣家信箱。
6. 系統將本次審核操作寫入後台操作日誌。

例外狀況—

> ➢ 賣家上傳之文件影像模糊或格式不符，管理員可於退回原因中說明，要求賣家重新上傳。
>
> ➢ 管理員執行退回操作時未填寫退回原因，系統提示「請填寫退回原因後再送出」。
>
> ➢ 審核列表頁面載入失敗，系統提示「資料載入失敗，請重新整理頁面」。

### 2-1 違規商品下架－事件條列式

| | |
|---|---|
| **二手買賣市場** | |
| **使用個案名稱** | 違規商品下架 |
| **行為者** | 管理員 |
| **目標** | 管理員對違反平台規範之商品執行強制下架，並通知賣家處理結果 |
| **前提** | 管理員已登入後台，且已確認特定商品具備違規事實（透過檢舉通報或主動巡查發現） |
| **結束狀態** | 商品狀態更新為「強制下架」，相關訂單已處置，通知已發送，操作已記錄 |

**一系列事件：**

正常程序—

1. 管理員於後台「商品管理」頁面，搜尋或篩選目標商品，或透過「檢舉管理」頁面直接跳轉至關聯商品。
2. 管理員進入商品詳情頁面，檢視商品資訊（名稱、描述、圖片、售價、賣家帳號）。
3. 管理員確認商品違規後，點擊「強制下架」按鈕。
4. 系統彈出確認視窗，要求管理員選擇違規原因類別（如：違禁品、虛假描述、侵權商品）並填寫說明（選填）。
5. 管理員確認送出，系統執行以下操作：
   - 更新商品狀態為「強制下架」。
   - 查詢與該商品相關之進行中訂單，標記為需人工介入。
6. 系統發送違規下架通知至賣家信箱，說明下架原因及申訴管道。
7. 系統將操作記錄寫入後台操作日誌。

例外狀況—

> ➢ 目標商品在管理員操作前已由賣家自行下架，系統提示「此商品已下架，無需再次操作」。
>
> ➢ 管理員未選擇違規原因即送出，系統提示「請選擇違規原因後再送出」。
>
> ➢ 通知信發送失敗，系統提示「通知信發送失敗，請至通知管理手動補發」。

### 2-2 異常價格偵測－事件條列式

| | |
|---|---|
| **二手買賣市場** | |
| **使用個案名稱** | 異常價格偵測 |
| **行為者** | 管理員（審核）、系統（自動偵測） |
| **目標** | 系統自動偵測平台上定價異常之商品（如：遠低於市價或惡意高價），並通知管理員進行人工審核 |
| **前提** | 商品已完成上架，系統異常價格偵測機制運作正常 |
| **結束狀態** | 異常商品已被標記並進入人工審核佇列，或自動完成處置 |

**一系列事件：**

正常程序—

1. 系統於商品上架時自動執行價格分析，比對「商品主檔」中同類別商品之歷史成交價與市場均價。
2. 系統依下列條件判斷是否為異常價格：
   - 定價低於同類商品市場均價 **X%** 以下（疑似惡意低價或詐騙）。
   - 定價高於同類商品市場均價 **Y%** 以上（疑似惡意哄抬）。
3. 若偵測到異常，系統自動將商品標記為「待審核（異常價格）」，並將商品加入後台「異常商品審核」佇列。
4. 系統發送異常通知至後台管理員通知中心，告知待審核商品數量。
5. 管理員進入「異常商品審核」頁面，檢視系統標記之商品清單。
6. 管理員點選特定商品進行審核（詳見 2-3 惡意價格處理）。

例外狀況—

> ➢ 系統因資料不足（如：新上架商品類別無歷史成交資料）無法進行比對，系統記錄「無法比對」並略過該商品，不列入異常佇列。
>
> ➢ 偵測機制執行失敗（如：資料庫連線異常），系統記錄錯誤日誌並發送警示通知至系統管理員。

### 2-3 惡意價格處理－事件條列式

| | |
|---|---|
| **二手買賣市場** | |
| **使用個案名稱** | 惡意價格處理 |
| **行為者** | 管理員 |
| **目標** | 管理員對系統標記之異常價格商品進行人工審核，並執行適當處置（警告、強制下架或解除標記） |
| **前提** | 商品已被系統標記為「待審核（異常價格）」，管理員已進入異常商品審核詳情頁面 |
| **結束狀態** | 管理員已完成審核判斷，相應處置已執行，賣家已收到通知，操作已記錄 |

**一系列事件：**

正常程序—

1. 管理員於「異常商品審核」頁面點選待審核商品，進入審核詳情頁面。
2. 系統顯示商品詳情（名稱、定價、類別、賣家帳號）、系統偵測依據（同類均價、偏差百分比）及賣家歷史評價。
3. 管理員進行人工判斷：
   - 若確認為**惡意定價**：
     - 選擇處置方式：「發送警告通知」或「強制下架」。
     - 填寫處置說明，確認送出。
     - 系統依選擇執行對應操作，並發送通知至賣家信箱。
   - 若判斷為**誤判（價格合理）**：
     - 點擊「解除標記」，系統移除異常標記，商品恢復正常上架狀態。
     - 系統不發送通知（或選擇性發送澄清通知）。
4. 系統將審核結果及處置操作寫入後台操作日誌。

例外狀況—

> ➢ 管理員審核期間商品已由賣家自行修改售價，系統提示「商品售價已變動，請重新確認後再執行處置」。
>
> ➢ 管理員未選擇處置方式即送出，系統提示「請選擇處置方式後再送出」。
>
> ➢ 通知發送失敗，系統提示「通知信發送失敗，請至通知管理手動補發」。

**一系列事件：**

正常程序—

1. 管理員進入後台「訂單監控」頁面，可依條件篩選訂單（如：訂單狀態、建立日期區間、賣家 / 買家帳號）。
2. 系統自「訂單主檔」查詢符合條件之訂單清單，顯示訂單編號、交易金額、目前狀態、買賣雙方帳號。
3. 系統以醒目方式標示「異常訂單」（如：超過出貨期限仍未出貨、已申請退款超過處理期限）。

5. 管理員依情況執行操作：
   - **催促賣家出貨**：系統發送系統通知至賣家，要求限時完成出貨。
   - **強制退款**：對確認無法完成交易之訂單，系統啟動退款流程，更新訂單狀態為「已退款」。
   - **標記人工介入**：對爭議訂單，標記為「人工介入中」，由管理員進一步調查。
   - **僅查看資訊**：不執行異動，直接離開頁面。
6. 系統將操作記錄寫入後台操作日誌。

例外狀況—

> ➢ 搜尋條件無符合訂單，系統提示「查無符合條件之訂單」。
>
> ➢ 執行強制退款時，若付款閘道回應失敗，系統提示「退款操作失敗，請稍後再試或手動聯繫金流服務商」。
>
> ➢ 訂單詳情頁面載入失敗，系統提示「資料載入失敗，請重新整理頁面」。

### 3-1 檢舉處理（後台）－事件條列式

| | |
|---|---|
| **二手買賣市場** | |
| **使用個案名稱** | 檢舉處理（後台） |
| **行為者** | 管理員 |
| **目標** | 管理員審核使用者提出之檢舉申請，判斷是否成立並執行對應處置，最後通知檢舉人審核結果 |
| **前提** | 使用者已完成檢舉申請（狀態為「審核中」），管理員已登入後台且具備檢舉審核權限 |
| **結束狀態** | 檢舉審核已完成，處置已執行，結果通知已發送，操作已記錄 |

**一系列事件：**

正常程序—

1. 管理員進入後台「檢舉管理」頁面，系統顯示所有狀態為「審核中」之檢舉申請清單（含檢舉類型、被檢舉對象、提交時間）。
2. 管理員點選特定檢舉申請，進入審核詳情頁面。
3. 系統顯示以下資訊：
   - 檢舉人帳號（匿名處理，僅顯示帳號 ID）。
   - 被檢舉對象（商品 / 評價 / 賣家）之詳細資訊。
   - 檢舉原因類別與補充說明。
   - 檢舉人上傳之證據（如有）。
4. 管理員審查後進行判斷：
   - 若**檢舉成立**：
     - 管理員選擇對應處置（如：刪除違規評價、對被檢舉方發送警告、執行商品下架或帳號停權）。
     - 確認送出，系統依處置類型執行相應操作。
     - 系統更新檢舉狀態為「檢舉成立」。
   - 若**檢舉不成立**：
     - 管理員填寫不成立原因（選填），確認送出。
     - 系統更新檢舉狀態為「檢舉不成立」。
5. 系統自動發送審核結果通知至檢舉人帳號（站內通知與 Email）。
6. 系統將本次審核操作寫入後台操作日誌，記錄審核管理員、時間戳記與處置方式。

例外狀況—

> ➢ 管理員審核期間，被檢舉商品已由賣家自行下架或刪除，系統提示「被檢舉對象已不存在，請確認後繼續審核」。
>
> ➢ 同一對象被多筆檢舉，系統提示「此對象尚有其他待審核檢舉，建議合併審查」，並提供快速跳轉連結。
>
> ➢ 審核詳情頁面載入失敗，系統提示「資料載入失敗，請重新整理頁面」。
>
> ➢ 執行關聯處置（如停權）時發生錯誤，系統提示「關聯操作執行失敗，請至帳號管理頁面手動處理」。
