# ADR-002: 檢索策略（Hybrid + Parent-Document）與「為何不做純向量 RAG」

**狀態**：Accepted  
**日期**：2026-04-18  
**決策者**：Student / RAG Architect  
**對應 Spec**：[Spec-006](../specs/spec-006-retrieval-pipeline.md)

> 期中格式對照：**Context**＝「脈絡與問題」；**Decision**＝「決策」；**Status**＝本文件抬頭狀態；**Consequences**＝「後果（含風險與緩解）」。

---

## 脈絡與問題

一般「純向量 RAG」在法律文本上會出現兩個結構性失敗模式：

1. **語意相近 ≠ 法源正確**  
   兩段文字可能在「損害賠償要件」語意上相近，但各自屬於不同請求權基礎或不同法域適用範圍。

2. **條號與字號是離散符號**  
   embedding 對「第 184 條」與「184」的表徵未必穩定；使用者也可能用錯誤條號提問，系統需要能「對齊到正確法源」而非「找最像的一段故事」。

因此，法律檢索需要 **關鍵字精準命中** 與 **語意召回** 並存，並在生成前以 **parent 還原** 提供足夠上下文以避免斷章取義。

---

## 決策

採用 **Hybrid Retrieval（BM25 + Dense）**，並加上 **Parent-Document Retrieval**，最後以 **cross-encoder rerank** 收口。

### 核心決策點

1. **BM25 承擔「法源座標」功能**  
   法規名、條號、專有術語、裁判字別等離散訊號，優先於關鍵字索引完成定位。

2. **Dense 承擔「爭點語意」功能**  
   使用者描述的事實與爭點，透過向量召回找到可能相關的判決要旨與函釋段落。

3. **Parent-Document 承擔「可解釋與不斷章取義」功能**  
   child 用於高召回；parent 用於把答案放回正確的法律語境。

4. **Fusion 採 RRF（MVP）**  
   避免不同路徑分數量級不一致造成的調參地獄；後續可改成加權融合（Phase 2）。

---

## 被拒絕的方案

### 方案 A：純向量 RAG

* **問題**：條號／字號命中率不稳；容易檢索到「像」但不可引用的段落。  
* **結論**：拒絕。

### 方案 B：只用 BM25（不做向量）

* **問題**：口語化爭點與類案推理弱；跨段落概念對齊差。  
* **結論**：拒絕。

### 方案 C：GraphRAG 作為 MVP 主路徑

* **問題**：法律關係邊的抽取成本高、錯邊會污染推理；期中交付風險大。  
* **結論**：MVP 拒絕；保留為 Phase 3 研究分支（見 [Spec-011](../specs/spec-011-roadmap-and-deliverables.md)）。

---

## 後果

### 正面影響

* 檢索結果更可用於「引用式回答」，提升可解釋性。
* 可針對不同路徑做消融實驗（A/B path contribution）。

### 負面影響與風險

* 系統複雜度上升：兩套索引、join、與更多監控點。
* BM25 與中文分詞／analyzer 設定錯誤會顯著影響法條命中。

### 緩解措施

* 以 `legal_unit_id` 強制 join Qdrant/OpenSearch。
* 建立「條號命中」小型測試集（30～50 題）作為 CI gate。
