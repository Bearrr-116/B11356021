# 法條檢索與案例輔助 RAG 白皮書

## 以「法源可追溯 + 混合檢索 + 證據化生成」為主軸的領域驅動架構

- Status: Draft
- Last Updated: 2026-04-18

---

## 1. 摘要

把「全國法規資料庫」與「裁判書公開資料」直接向量化後丟進 LLM，並不足以構成可用的法律資訊系統。法律場景的核心難題不是「找得到一段很像的文字」，而是：

> **找得到正確法源座標（條號／版本），並能把類案要旨放回正確的構成要件脈絡，且全程可追溯。**

因此，本白皮書主張：法律 RAG 應採 **Hybrid Retrieval + Parent-Document Reconstruction + Reranking + Grounded Generation**，並以 **Answer Contract（結構化、可測試）** 作為生成前的強約束介面。

本文件對應之可實作規格，已拆為 `docs/specs/spec-001`～`spec-011`；架構決策見 `docs/adr/ADR-001`～`ADR-003`。

## 2. 問題定義：法律檢索不是「相似度搜尋」

### 2.1 使用者查詢的典型樣態

法律使用者常同時混合：

* **座標式線索**：「民法 184」「消保法第 12 條」
* **敘事式線索**：網購糾紛、車禍、勞資爭議、契約解除等故事描述
* **不確定線索**：不確定管轄、不確定程序階段、不確定契約定性

### 2.2 一般 RAG 的典型失敗

1. **語意相似但法源錯誤**  
   兩段文字可能都在討論「損害賠償」，但請求權基礎不同、構成要件不同、舉證責任分配不同。

2. **條號與版本敏感**  
   法條會修；同一條號在不同版本可能對應不同文字。若索引不版本化，會出現「引用看似正確但其實過期」的錯覺。

3. **判決摘要的敘事噪音**  
   判決文本含大量程序與事實背景；向量檢索容易被「故事像」誤導，而非「法理像」。

4. **可引用性不足**  
   教學或研究輔助需要「可回到原始法源與案件識別資訊」。若輸出只有自然語言而無 citation graph，難以稽核。

## 3. 系統定位與責任邊界

### 3.1 系統目標

* **法條檢索**：把問題對應到可信條文與子法體系
* **案例輔助**：提供判決摘要中與爭點相關的要旨整理（非勝敗預測）
* **可追溯回答**：每一次法律概念推進，盡量對應到 `legal_unit_id`

### 3.2 非目標（必須明示）

* 不提供個案正式法律意見
* 不替代律師進行策略選擇
* 不根據不完整程序資訊做出確定結論

此邊界與 [Spec-001](../specs/spec-001-system-overview.md)、[Spec-005](../specs/spec-005-jurisdiction-rule-engine.md) 一致。

## 4. 核心設計原則

### 原則一：chunk 邊界服從法律文本結構

法律文本天然是階層結構（編章節、條項款目）。固定長度切分會造成：

* 要件被切成上下無關的片段
* 「但書」與「本文」分離
* 引用顯示時無法呈現完整語境

因此本系統採 **parent-child**：child 用於召回，parent 用於還原與展示（見 [Spec-004](../specs/spec-004-legal-unit-schema.md)）。

### 原則二：先正規化查詢，再檢索

法律檢索的前置問題是「把使用者語言轉成可檢索座標」：

* 法規別名解析（消保法 → 消費者保護法）
* 條號提示抽取（民法 184）
* BM25 與 Dense 各自最佳 query 分流（見 [Spec-002](../specs/spec-002-legal-query-normalizer.md)）

### 原則三：混合式檢索是必要條件，不是加分項

* BM25 擅長 **離散符號** 與 **專有術語**
* Dense 擅長 **爭點語意** 與 **跨段落概念對齊**

因此採 Hybrid，並以 RRF 作為 MVP 融合策略（見 [Spec-006](../specs/spec-006-retrieval-pipeline.md)）。

### 原則四：生成必須證據化（Grounded）

本系統以 Stage 1 Answer Contract 作為「可驗證的中間表示」，Stage 2 僅允許在 citations 範圍內敘事（見 [Spec-007](../specs/spec-007-generation-orchestrator.md) 與 ADR-003）。

## 5. 知識表徵的理論難點（Representation）

### 5.1 「法律概念」不是單一向量

同一概念在不同制度脈絡有不同效果。向量空間中的鄰近性，最多反映「語料共現與語言模式」，不足以表徵：

* 構成要件是否該當
* 抗辯是否成立
* 程序要件是否具備

因此需要 **法源座標**（條號 + 版本）與 **證據片段** 的共同表徵，而不是單一 embedding。

### 5.2 文本與規範命題的落差

判決摘要常是「自然語言敘述」，而法條是「規範命題」。兩者在字面上可能不一致，但在法理上互相引用。

此落差導致：

* 僅用判決 embedding 可能漏掉條文核心
* 僅用條文 embedding 可能漏掉實務限縮或類型化判準

本系統以 **metadata tags（issue tags）** 與 **Hybrid** 緩解，並保留 GraphRAG 作為 Phase 3 研究方向（見第 8 節）。

## 6. Hallucination（幻覺）治理：把「不可信生成」轉成可觀測指標

### 6.1 三層治理

1. **檢索層**：Hybrid + parent + rerank，降低不相干片段進入生成。  
2. **規則層**：程序／時效／證據不足旗標，直接切換 `generation_mode`。  
3. **生成層**：Answer Contract + 引用守門 +（可選）後處理檢查。

### 6.2 評估層：RAGAS + TruLens

* RAGAS 提供可批次計算的 `faithfulness` / `context_*` 指標（見 [Spec-009](../specs/spec-009-evaluation-framework.md)）
* TruLens 提供 `groundedness` 與線上監控能力

法律域建議補充 **citation integrity**（引用是否真能對應到證據列表）。

## 7. Cold Start 與專有名詞對齊（Cold Start / Terminology）

### 7.1 Cold Start：資料稀疏時怎麼開始

建議採 L0 法條核心集合先行（權威、結構清楚），再逐步加入函釋與判決摘要：

* L0 建立「條文 parent-child」骨架
* L1 函釋補齊適用爭議
* L2 判決摘要補齊實務類型化判準

此與 [Spec-003](../specs/spec-003-knowledge-etl.md) 的分層一致。

### 7.2 專有名詞對齊（Alias / Normalization）

法律語言存在大量：

* 簡稱（消保法）
* 半專業口語（車禍賠償）
* 中英混用（GDPR 風格敘述）

需要 **curated alias table** + **查詢正規化器** 共同完成對齊，而不是只靠 embedding「自己學會」。

---

## 8. 為何期中 MVP 不採 GraphRAG 作主路徑？

GraphRAG 的價值在於顯式表達「引用／遵循／區辨」等關係，特別適合研究型探索。但其成本在於：

* 關係抽取錯誤會形成錯誤推理鏈
* 建圖與維運複雜度高
* 需要額外品質治理與版本治理

因此本專案 **MVP 以 Hybrid + parent + rerank 為主**；GraphRAG 保留為 Phase 3 研究分支（見 [Spec-011](../specs/spec-011-roadmap-and-deliverables.md) 與 ADR-002）。

---

## 9. 風險、限制與合規

* 法規版本錯置風險：必須 `law_version_id` 治理
* 個資風險：查詢與文本去識別化（Spec-002 / Spec-003）
* 模型漂移：每次升級必跑評估基準（Spec-009）

---

## 10. 結論

法律領域驅動 RAG 的關鍵不在「更大的模型」，而在「更清楚的法源座標與證據鏈」。  
本專案以可實作 spec 拆分模組責任，並以評估框架把品質變成可量測的工程問題，使期中規劃書同時具備 **學理深度** 與 **工程可交付性**。
